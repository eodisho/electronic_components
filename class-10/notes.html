Time

You only have a limited about of time. How can you not waste it?"

Time is limited when dealing with something else with a timer. In this case, lets use a human. A human has a visual refresh rate of about 24Hz, which means anytthing happening slower than 24Hz will be noticed, and anything happening faster won't be. For example, animation looks smooth above 24Hz, typically referred to as frames per second.

So, in this case, what's a frame? This is anything you need to do before the human notices that something has changed. Simple example: an LED is pulsing, how you can detect a button click?

loop() { for (x =0, x<=255, x++) { analogWrite(pin, x); delay(100); } y = digitalRead(button); if (y == HIGH) { doSomething(); // this takes 50ms } }
How long will this take to run?
  • Will the button presses always be detected?
  • What's going to happen to the LED pulses?
The problem is that the for look will have to complete before checking the button. And checking the button is so fast, we're guaranteed to miss it unless it's held down. wat do?
The key is to manage your time effectively. delay statements literally do nothing, nothing else can run while they are running. If we can still pulse the led with 100ms intervals, but we don't have to block the rest of the execution, we acquire 25500 ms to do other stuff....or do we?
We still need to update the led every 100ms though. But 100ms still frees up time to check the button. We'll use waits and delays.
int wait = 0; int delay = 100; loop() { if (millis - wait > delay) { x +=1 analogWrite(pin, x); wait = millis(); } y = digitalRead(button); if (y == HIGH) { doSomething(); } }
Not only is this non-blocking, but a foor loop was eliminated. remember, loop is already a loop! use it to your advantage. Are we in the clear?
